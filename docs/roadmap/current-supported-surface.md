## 一页结论

- Phase 1 到 Phase 6 的编译器和 same-file tooling 已经进入稳定迭代阶段。
- Phase 7 已有保守 async/runtime/build 子集：`async fn`、`await`、`spawn`、`for await`、最小 `ql-runtime`、program-mode async `main`、保守 async `staticlib` / `dylib` 子集，以及 task-handle-aware lowering。
- 但 Phase 7 的 async/codegen 面仍在持续回归收口；如果文档条目与当前实现不一致，以 `crates/ql-codegen-llvm/src/tests/*`、`crates/ql-cli/tests/codegen.rs` 和实际测试结果为准。
- Phase 8 已进入真实交付面：最小 `qlang.toml` package/workspace graph、第一版 `[dependencies]` 本地路径依赖输入、`ql project init`、`ql project targets`、`ql project lock`、`ql run`、`ql test`、`.qi` V1 emit/load、`ql project graph`、`ql project emit-interface`、`ql build --emit-interface`、`ql check --sync-interfaces`。
- project-aware `ql build` / `ql run` / `ql test` 现在已能穿过一条真实依赖执行路径：root package 可以调用 direct dependency 暴露的 public `extern "c"` 符号，并在 executable/test 路径上复用预构建的 dependency staticlibs。
- dependency-backed cross-file tooling 已开放首批合同：imported dependency symbol hover / definition / declaration / references、import path completion、dependency enum variant completion / `typeDefinition`、显式 struct field-label completion / query / `typeDefinition`（现在也覆盖 `Cfg.Scope.Config { child: value }` 这类更深层 struct-like path），以及语法局部可恢复 receiver 的最小 dependency member/query/typeDefinition；当同包里存在无关 dependency `.qi` 缺失、损坏、不可读，或某个 referenced package manifest 自身损坏时，这些 package-aware dependency import/completion/definition 基础能力现在也会继续保留其余健康依赖的结果，不再整包掉空。
- 保守 `workspace/symbol` 现在可以在有 manifest 上下文时搜索当前包源码、同一 workspace 的 sibling members、当前包依赖 `.qi` 的 public symbols，以及 sibling member 依赖 `.qi` 的 public symbols；这里的 dependency public symbols 也明确包括 public trait / `impl` / `extend` methods，例如 `Config.get`、`Reader.poll`、`Buffer.twice`；当前包依赖与 sibling member 依赖两侧现在都已对 `trait` / `impl` / `extend` 三类 methods 建立显式 regression 锁定；对于当前包依赖侧与 sibling member 依赖侧，`missing .qi`、`invalid reference manifest` 与 `source diagnostics` 三条主要 fallback 入口下的 public method 保留面现在也都已有独立回归保护，而且 `trait` / `extend` methods 也不再只靠通用 method 回归隐含覆盖；如果当前包或 sibling member 因源码 diagnostics、单个 dependency `.qi` 缺失、损坏或不可读而无法整包分析，也会回退到 manifest/source + 可用依赖接口的粒度，继续保留仍可单独分析的模块符号和其余健康依赖符号，而不是因为一个坏依赖把整条搜索面一起丢掉；另外，当 `qlsp` 从 editor initialize 请求拿到 workspace roots 时，即使没有打开任何 `.ql` 文档，也会从 workspace manifest/source 建立这条保守 `workspace/symbol` 搜索面。
- package/workspace import 的 definition / declaration 现在会优先尝试跳到当前 workspace 里唯一可定位的源码定义；只有找不到唯一源码目标时，才回退到 dependency `.qi` 里的接口定义位置。这条能力当前仍是保守导入面，不等于任意跨文件语义都已开放。
- `qlsp` 现在也补上了 `textDocument/documentHighlight`：同文件符号家族会直接复用 same-file / package-aware references 真相面，把当前文件内的定义和使用位统一高亮；这条能力目前仍限制在“当前文档内高亮当前文档 occurrence”，不扩大成 project-scale semantic highlighting。

## 当前已开放的构建表面

### 前端与 same-file tooling

- lexer、parser、formatter、diagnostics、HIR、resolve、typeck、MIR、borrowck 已进入主路径。
- same-file hover、definition、references、document highlight、rename、completion、semantic tokens、document symbol 已接通到共享 analysis/query surface。
- same-file rename 现在也会把更深层 struct-like 简写绑定（如 `Point.Scope.Config { x }`）改写成显式绑定（如 `Point.Scope.Config { x: coord_x }`）；同一层上的显式字段标签（如 `Point.Scope.Config { x: value }`）也已进入 same-file field hover / definition / references / rename / semantic tokens；更深层 struct-like literal / pattern 路径里的字段标签补全（如 `Point.Scope.Config { fl: value }`）和字段 `typeDefinition`（如 `Point.Scope.Config { x: value } -> Coord`）现在也会直接收敛到同一 field surface。

### project / workspace / `.qi`

- package directory、`qlang.toml`、包内源码路径、workspace-only 根 manifest 都可进入 package-aware `check` 流程。
- 当前 manifest 已开放的字段仍然很少：`[package].name`、`[workspace].members`、path-only 的 `[lib].path` / `[[bin]].path`、legacy `[references].packages`、新的 `[dependencies]` 本地路径依赖写法，以及第一版 `[profile].default`；其中 `[lib]` / `[[bin]]` 当前只支持 `path = "src/..."`，并要求目标文件是 package 内 `src/` 下的 `.ql` 源码；`[dependencies]` 当前只接受 `dep = "../dep"` 或 `dep = { path = "../dep" }`；`[profile]` 当前只接受 `default = "debug"` 或 `default = "release"`。如果 profile 写在 package manifest 上，它会先作为该 package 自己的默认 profile；如果 profile 写在 workspace 根 manifest 上，它会作为 member 的缺省 profile，并且仅在 member 自己没有声明 `[profile].default` 时生效。这两种依赖写法在内部都会先归一到现有 `references.packages` 列表，因此今天的 `ql project graph`、`.qi` 生命周期、`ql check --sync-interfaces`、dependency-backed analysis/LSP 仍共享同一条 reference/package truth surface；此外，workspace 级 `ql project targets` / `ql build` / `ql run` / `ql test` 现在也会按这些本地依赖对 workspace members 做依赖优先排序，并在 workspace members 之间存在本地依赖环时显式失败。version、registry、feature，以及更完整的 profile 配置/继承仍未开放。
- `ql project init` 已可直接生成最小 package / workspace 骨架；workspace 模式默认生成根 `qlang.toml`、`packages/<name>/qlang.toml`、`packages/<name>/src/lib.ql`、`packages/<name>/src/main.ql` 与 `packages/<name>/tests/smoke.ql`，单 package 也会同步生成 `src/main.ql` 与 `tests/smoke.ql`，因此新脚手架现在可以直接进入 `ql project graph` / `ql check` / `ql build` / `ql run` / `ql test`。
- `ql project targets` 已可枚举 package/workspace 的共享 project target 发现面：默认约定仍是 `src/lib.ql`、`src/main.ql`、`src/bin/**/*.ql`，但现在也支持 path-only 的 `[lib].path` 与 `[[bin]].path` 显式声明；若声明存在，对应 target family 会优先使用显式路径而不再回退到默认约定。若没有约定入口但包内只有一个源码文件则回退为 `source`；若没有可识别 target，会显式输出 `targets: (none found)`。同时，`ql project targets --json` 现已提供第一条稳定的机器可消费 project 输出，schema 为 `ql.project.targets.v1`，会按依赖顺序输出 members，并把每个 target 收敛成 `kind + path`。这组 target 现在同时服务 project-aware `ql build` 与 `ql run`。
- `ql project lock` 现已提供第一版 lockfile / reproducible-input surface：会在当前解析到的 package 或 workspace manifest 根目录写出 `qlang.lock`，schema 为 `ql.project.lock.v1`。首版 lockfile 当前锁定的是 resolved local package graph，而不是 registry/version 求解结果：会导出 root kind/manifest、workspace member roots（如果当前 root 是 workspace）、按确定性依赖顺序展开的 packages，以及每个 package 的 `manifest_path`、`package_name`、`selected`（是否是本次 root package 或 workspace member）、effective `default_profile`、resolved local dependency manifest paths 与当前 target surface（`kind + path`）。`ql project lock --check` 会在不改写文件的前提下校验现有 `qlang.lock` 是否仍与当前 manifest/dependency/target/profile 输入面一致；缺失或 stale lockfile 会显式失败，并提示直接重跑 `ql project lock <manifest>`。当前这条 lockfile 还不覆盖源码 hash、artifact、`.qi` 状态、registry/version/features 或更完整的 profile 模型。
- `ql build` 现在也接受 package directory、`qlang.toml`、workspace directory 与 workspace-only 根 manifest；会按 `ql project targets` 的发现结果逐个构建 target，并把默认产物写到各 package 自己的 `target/ql/<profile>/` 下，而不是统一堆到调用目录。默认约定 target 仍会保留 `src/` 之后的相对目录层级，例如 `src/bin/tools/repl.ql -> target/ql/debug/bin/tools/repl.ll`；显式 target 声明也走同一套规则，例如 `[[bin]] path = "src/tools/repl.ql" -> target/ql/debug/tools/repl.ll`。当前也已支持最小 target selector：`--package <name>` 可先收窄到 workspace member，`--lib|--bin <name>|--target <path>` 可继续收窄到单个 target；一旦 selector 把结果收窄到单 target，`--output` / `--header-output` 也就可以继续使用。没有 selector 时，多 target 场景仍显式禁止 `--output` 与 `--header-output`，避免单一路径参数污染整组 target；如果多个 target 会落到同一个默认 artifact 或 header 输出路径，CLI 也会先显式失败。`ql build` 现在还会把 manifest 里的本地 path dependency package 递归纳入构建计划：先按依赖优先顺序构建上游 package，再构建被用户选中的 root package；workspace member 会继续复用 workspace 级 profile 继承与排序，workspace 之外的本地依赖 package 则会按它自己的 manifest 默认 profile 落到各自包根下的 `target/ql/<profile>/`。`ql run` / `ql test` 这轮也已接上同一套依赖预构建图，会在真正执行 root runnable target 或 smoke tests 之前先静默构建上游本地依赖 package；同时 root target 已打开一条保守的依赖执行例外，可通过 direct dependency 的 public `extern "c"` 符号完成构建/执行。除此之外，普通跨包 Qlang 语义仍未开放。`--emit-interface` 现在允许在 package/workspace build 上使用，并且会在每个 package 的全部 discovered targets 构建成功后只发射一次默认 `.qi`。如果这次调用没有显式传 `--emit`，`lib` target 会保守落成 `staticlib`，`bin` / fallback `source` target 继续沿当前 single-file 默认 emit；一旦显式传了 `--emit`，整次 project-aware build 仍共享同一种 artifact 形态。对于 package/workspace 路径，如果 CLI 没有显式传 `--profile` 或 `--release`，构建会先尝试目标 package 自己的 `[profile].default`，再回退到 workspace 根 manifest 的 `[profile].default`；一旦显式传了 `--profile <debug|release>` 或 `--release`，CLI 选择会优先覆盖 manifest 默认值。
- `ql build --json` 现已开放第一版机器可消费构建输出，schema 为 `ql.build.v1`。当前稳定边界已覆盖成功构建结果，以及六类 `status: "failed"` 结果：project-aware preflight failure（当前已覆盖 `project-context`、`manifest-load`、`target-discovery`、`target-selection`、`output-planning`，以及 selected package 无 discovered targets 的 `build-plan` 校验）、build-plan recursion failure（当前稳定继续落到 `stage = build-plan`，并已覆盖 local dependency cycle 与递归 dependency target-discovery / manifest failure）、dependency/interface prep failure（当前稳定落到 `stage = dependency-interface-prep`）、per-target target-prep failure（当前稳定落到 `stage = target-prep`，已覆盖 direct dependency extern 冲突，以及依赖 manifest / interface artifact / target source 读取这类更深的准备阶段失败）、build-side `emit-interface` failure，以及 actual target build failure。JSON 会导出请求路径、`scope`（`file` / `project`）、resolved `project_manifest_path`、请求的 `emit/profile`、`profile_overridden`、`emit_interface`，以及按实际构建顺序展开的 `built_targets` 与 `interfaces`；成功时 `failure = null`，失败时 `failure` 会定位到 preflight failure point、build-plan recursion failure point、dependency/interface prep failure point、per-target target-prep failure point、build-side interface emit failure point 或失败 target。actual target build failure 继续使用 `diagnostics` / `invalid-input` / `io` / `toolchain` 这四类 `error_kind`；preflight failure、build-plan recursion failure、dependency/interface prep failure、per-target target-prep failure 与 build-side interface emit failure 则额外稳定导出 `stage`，并按场景补 `selector`、`conflict_path`、`target_count`、`output_path`、`source_root`、`failing_source_count`、`first_failing_source`、`owner_manifest_path`、`dependency_manifest_path`、`dependency_package`、`interface_path`、`reference_manifest_path`、`reference`、`failing_dependency_count`、`first_failing_dependency_manifest`、`cycle_manifests`、`symbol`、`first_dependency_package`、`first_dependency_manifest_path`、`conflicting_dependency_package`、`conflicting_dependency_manifest_path`、`io_path` 等 detail。`built_targets` 当前稳定收敛成 `manifest_path`、`package_name`、`selected`、`dependency_only`、`kind`、`path`、`emit`、`profile`、`artifact_path`、`c_header_path`；`interfaces` 当前稳定收敛成 `manifest_path`、`package_name`、`selected`、`status`（`wrote` / `up-to-date`）与 `path`。project-aware build 在进入 root target 之前若需要递归同步本地依赖 `.qi`，这类前置同步写出现在也会继续保持 stdout 静默，不再把 `wrote interface: ...` 混进 JSON 输出；之前剩下的 per-target dependency extern/interface artifact prep 这块失败面，现在也已稳定留在 stdout JSON contract 内，而不会再绕回 stderr-only 退出。
- `ql run` 现在接受单文件、package directory、`qlang.toml`、workspace directory 与 workspace-only 根 manifest；单文件路径会直接按 `exe` build 并执行产物，package/workspace 路径则复用 `ql project targets` 的发现结果。当前 runnable target 仅包括 `bin` 和 fallback `source`，`lib` 明确不会参与 `run`；默认情况下仍要求全局恰好只有一个 runnable target，但现在也已支持 `--package <name>`、`--bin <name>` 与 `--target <path>` 来在多 target 项目里选中唯一 runnable target，而显式 `[[bin]].path` 也会直接进入同一套 runnable target 发现面。project-aware `ql run` 这轮还会在真正构建 root executable 之前，静默同步 root package 所需的本地依赖 `.qi` 并静默预构建该 package 的本地 path dependency 闭包；这些 dependency-only package 会落到各自默认 artifact 路径，不会污染程序 stdout/stderr。`--profile <debug|release>` 与 `--release` 会显式选择 executable build profile；如果 project-aware `ql run` 没有传这些 CLI 选项，则会先回退到目标 package manifest 里的 `[profile].default`，再回退到 workspace 根 manifest 的 `[profile].default`。`-- <args...>` 会原样转发给最终可执行文件。
- `ql test` 现在提供第一版 project-aware 测试 harness：单文件路径仍会把该 `.ql` 文件按 executable 构建并执行；package/workspace/`qlang.toml` 路径则会递归发现各 package 下的 `tests/**/*.ql`。其中普通 `tests/**/*.ql` 仍按 executable smoke test 构建并执行，而 `tests/ui/**/*.ql` 现在会走第一版项目内 UI snapshot 合同：直接按单文件语义分析运行，要求产生 diagnostics，并与相邻 `.stderr` 快照做精确比对。`--package <name>` 会先把 discovery 收窄到单个 package/workspace member，再继续进入同一套 smoke / UI、`--list`、`--filter` 与 `--json` 流程。project-aware `ql test` 这轮也会在真正执行 smoke tests 之前静默预构建这些测试所属 package 的本地 path dependency 闭包；dependency build 消息不会混入正常测试输出，而 UI snapshot tests 仍只走 analysis/diagnostics。对 package/workspace 路径，如果 CLI 没有显式传 `--profile` 或 `--release`，普通 smoke tests 的 build profile 会先回退到目标 package manifest 里的 `[profile].default`，再回退到 workspace 根 manifest 的 `[profile].default`；显式 CLI profile 会优先覆盖 manifest 默认值。没有发现任何测试文件时 CLI 会显式失败，并提示补 `tests/**/*.ql` 或直接指定单个 `.ql` 文件；如果 discovery 成功但过滤后没有匹配项，也会显式失败并建议先重跑 `--list`；workspace member 没有测试时当前会保守跳过，而不是让整次 `ql test` 直接失败。这个 UI 模型目前仍只覆盖单文件 diagnostics snapshot，不等于完整 integration/doc/benchmark harness 已开放。
- `ql test --json` 现已开放第一版机器可消费测试输出，schema 为 `ql.test.v1`。当前稳定边界覆盖 `status: "listed" | "ok" | "failed" | "no-tests" | "no-match"` 五类结果，并稳定导出 `path`、`requested_profile`、`profile_overridden`、`package_name`、`filter`、`list_only`、`discovered_total`、`selected_total`、`targets`、`passed`、`failed`、`failures` 等字段；其中 target 当前收敛成 `path + kind (+ effective smoke profile)`，failure 当前收敛成 `build` / `run` / `spawn` / `ui` 四类。manifest 加载失败、package selector/path 失败、dependency/interface 预备失败等更早阶段的错误当前仍沿既有 stderr 合同，不在这条 JSON slice 里假装已经完全结构化。
- `ql project graph` 会展示 package/member、references，以及默认 `.qi` 的 `valid` / `missing` / `invalid` / `stale` 状态；`stale` 会给出 `stale_reasons`，`invalid` / `unreadable` 也会给出一行 `detail`；这些 `detail` / `member_error` 字段里的 manifest 路径现在也统一按 graph 根目录做相对化，不再混入绝对路径；每条 `reference_interfaces` 现在也会显式带出对应的 reference manifest 路径，方便直接定位引用项；如果 direct dependency 下面还有更深层坏引用，也会补一个保守的 `transitive_reference_failures` 计数，以及第一个坏点的 manifest / `.qi` path / status / detail；当这个坏点本身是 `stale` 时，也会继续带出第一处 transitive stale 的 `stale_reasons`。同时，`ql project graph --json` 现已提供第二条稳定的机器可消费 project 输出，schema 为 `ql.project.graph.v1`，会把 package/workspace graph、interface 状态、reference 状态和 workspace member 展开结果一起导出。
