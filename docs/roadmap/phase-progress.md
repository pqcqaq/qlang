## 总体结论

- Phase 1 到 Phase 6 的基础设施已经落地：lexer、parser、formatter、diagnostics、HIR、resolve、typeck、MIR、borrowck、LLVM build、CLI 与 same-file LSP/query 都在代码里持续演进。
- Phase 7 已有可用子集：`async fn`、`await`、`spawn`、`for await`、最小 `ql-runtime`、task-handle lowering、保守的 program/library async build surface。
- Phase 8 已进入真实交付面：`qlang.toml` package/workspace loader、第一版 `[dependencies]` 本地路径依赖输入、`ql project init`、`ql project graph`、`ql project lock`、`ql run`、`ql test`、`.qi` emit/load、`ql build --emit-interface`、`ql check --sync-interfaces`、首批 dependency-backed cross-file tooling，以及 project-aware `build/run/test` 下 direct dependency public `extern "c"` 调用这条窄执行路径。
- 当前主线是沿既有边界扩面，不是回头重写前六阶段。
- 近期优先级顺序按开发计划执行：先补 package/workspace 的真实项目工作流闭环，再补 manifest / 依赖与基础 LSP 工程化，把跳转、高亮、workspace symbol 做到真实项目可依赖；可复现构建和更高级 IDE/refactor 能力随后跟进；新的 async/runtime/语言扩面继续后置。
- 真相源以 `crates/*`、`crates/ql-cli/tests/*`、`crates/ql-lsp/tests/*`、`crates/ql-analysis/tests/*` 和 [当前支持基线](/roadmap/current-supported-surface) 为准。

## 各阶段一句话总结

- Phase 1：把源码稳定变成 AST，并保证 formatter 与基础 CLI 可用。
- Phase 2：把“能解析”推进到“能做名称解析、类型检查和最小 IDE 查询”。
- Phase 3：建立 MIR 和 ownership/cleanup 分析，为后续 lowering 提供稳定中层。
- Phase 4：打通 `ql build` 和 LLVM backend，让受支持子集可以产出真实 artifact。
- Phase 5：把 C ABI 固定成当前稳定互操作边界，并生成 header sidecar。
- Phase 6：把 same-file 编辑器语义收口到共享 analysis/query truth surface。
- Phase 7：继续扩 async/runtime/task-handle/build/interop，但只开放已被语义、borrowck、codegen 和黑盒回归共同证明的子集。
- Phase 8：把 package/workspace、`.qi` 和 dependency-backed tooling 做成共享边界，避免 CLI 和 LSP 各写一套依赖模型。

## 当前进度对账

- 编译器主路径稳定为 AST -> HIR -> resolve -> typeck -> MIR -> LLVM IR。
- CLI 当前已实现 `ql check`、`ql fmt`、`ql mir`、`ql ownership`、`ql runtime`、`ql build`、`ql run`、`ql test`、`ql project`、`ql ffi`；其中 `ql project init` 已可直接生成最小 package / workspace 骨架，并同步生成 `src/main.ql` 与 `tests/smoke.ql`；`ql build --emit` 当前稳定面为 `llvm-ir|asm|obj|exe|dylib|staticlib`。
- `ql project targets` 已落地为真实 CLI 能力，可枚举 package/workspace 的 `lib` / `bin` / fallback `source` targets，当前已经直接服务 project-aware `ql build` / `ql run`；这轮 manifest 也补上了第一版 path-only target declarations：`[lib].path` 与 `[[bin]].path` 现在可把 target 显式指到 package 内 `src/` 下的任意 `.ql` 文件，并覆盖对应的默认约定发现面。与此同时，`ql project targets --json` 与 `ql project graph --json` 也已补成第一批稳定的机器可消费 project 输出，schema 分别为 `ql.project.targets.v1` 与 `ql.project.graph.v1`，可直接给脚本、CI 和后续编辑器集成消费同一套 target / graph 真相面。`ql project lock` 这轮也已补出第一版 `qlang.lock`：会把当前 root manifest、workspace member roots（如果当前 root 是 workspace）、按确定性依赖顺序展开的 resolved local package graph、effective default profile，以及每个 package 的 target surface 固化到 `ql.project.lock.v1`；`ql project lock --check` 则会在不改写文件的前提下把缺失或 stale 的 `qlang.lock` 显式挡住。`ql check --json` 这轮也补出了第一版机器可消费 diagnostics 输出，schema 为 `ql.check.v1`；当前稳定边界只覆盖 success 与 source diagnostics 两类结果，manifest/reference/interface artifact failure 仍保留既有 stderr 合同。`ql build --json` 这一轮也补出了第一版机器可消费构建输出，schema 为 `ql.build.v1`；当前稳定边界已覆盖成功构建结果、project-aware preflight failure（当前已覆盖 `project-context` / `manifest-load` / `target-discovery` / `target-selection` / `output-planning`，以及 selected package 无 discovered targets 的 `build-plan` 校验）、build-plan recursion failure、dependency/interface prep failure、per-target target-prep failure、build-side interface emit failure，以及 actual target build failure，并按实际构建顺序导出 root target、dependency-only target、partial built_targets 与 interface emit 记录。`ql.test --json` 这一轮也补出了第一版机器可消费测试输出，schema 为 `ql.test.v1`；当前稳定边界覆盖 `listed / ok / failed / no-tests / no-match` 五类结果，而 manifest/path/dependency-interface 这类更早失败面仍保留既有 stderr 合同。`ql test` 当前则已补到第一版 project-aware harness：普通 `tests/**/*.ql` 走 executable smoke，`tests/ui/**/*.ql` 走项目内单文件 diagnostics snapshot。
- `ql project lock` 这轮已补成第一版 lockfile workflow：package/workspace 根目录现在可以直接写出 `qlang.lock`，也可以用 `ql project lock --check` 校验现有 lockfile 是否 stale。当前 schema 为 `ql.project.lock.v1`，会导出 root kind/manifest、workspace selected members、transitive local dependency closure、effective default profile，以及每个 package 当前发现到的 targets；范围仍只覆盖本地 path dependency + 当前 target/profile 模型，不包含 registry/version/feature 求解。
- manifest 这轮先补了第一版本地依赖输入面：除了 legacy `[references].packages` 之外，`[dependencies]` 现在也接受 `dep = "../dep"` 与 `dep = { path = "../dep" }`；同时也补上了第一版 build profile 输入：`[profile].default = "debug" | "release"` 现在既可以写在 package manifest 上，也可以写在 workspace 根 manifest 上。workspace 根 profile 会作为 member 的缺省 profile，且只在 member 自己没有声明 `[profile].default` 时生效；project-aware `ql build` / `ql run` / `ql test` 在未显式传 CLI profile 时，也会按 `CLI override > package profile > workspace profile > debug` 的顺序取值。这些输入当前仍会先归一到既有 `references` / `.qi` 模型或最小 build-profile 决策面，因此 CLI、driver、analysis、LSP 这一轮不需要同时切换到第二套依赖图；不过 workspace 级 `ql project targets` / `ql build` / `ql run` / `ql test` 现在已经开始真实消费本地依赖关系，用它来决定 workspace member 的执行顺序，并在 members 之间形成局部依赖环时直接拒绝继续。version、registry、feature，以及更完整的 profile 配置仍未开放。
- `ql build` 这轮已接到同一套 project target 发现面：package/workspace 路径现在会逐个构建已发现 target，默认产物按各 package 根目录各自落到 `target/ql/<profile>/`，不再把 workspace member 的默认输出混到同一个根目录；`src/bin/...` 这类 nested target 也会把相对目录层级带到产物路径里。多 target 场景默认仍显式拒绝 `--output` 与 `--header-output`；但现在如果通过 `--package <name>`、`--lib|--bin <name>|--target <path>` 把结果收窄到单个 target，这两类单路径参数又可以继续使用。`ql build` 还首次把 manifest 本地 path dependencies 纳入 package 级构建图：CLI 会递归收集 root package 的本地依赖闭包，先构建依赖包，再构建被选中的 root package；workspace member 继续沿用 workspace profile 继承，workspace 外部的本地依赖包则按自己的 manifest profile / 默认 debug 落产物。`ql run` / `ql test` 这轮也已消费同一套 package build-prep graph，会在真正执行 root runnable target 或 smoke tests 之前静默预构建上游本地依赖 package；同时 project-aware root target 现在已经打开一条窄代码生成路径，可通过 direct dependency 的 public `extern "c"` 符号完成真实跨包调用与链接。但这仍不等于完整的 dependency-aware backend，普通跨包 Qlang free function/member/const 语义还没有开放。`--emit-interface` 现在允许在 package/workspace build 上使用，并且会在每个 package 的全部 discovered targets 构建成功后只发射一次默认 `.qi`。另外，如果这次调用没有显式传 `--emit`，`lib` target 会保守走 `staticlib`，`bin` / fallback `source` target 继续沿当前 single-file 默认 emit；一旦显式传了 `--emit`，整次 project build 仍共享同一种 artifact 形态。`ql build` 现在也已补上 `--profile <debug|release>`；对于 package/workspace 路径，如果 CLI 没有显式传 `--profile` 或 `--release`，则会按 `package profile -> workspace profile -> debug` 的顺序解析默认 profile。`ql.build.v1` 现在会把成功构建、project-aware preflight failure、build-plan recursion failure、dependency/interface prep failure、per-target target-prep failure、build-side interface emit failure 和 actual target build failure 一起稳定导出：成功时给出完整 built_targets / interfaces；preflight failure 会定位到 `project-context` / `manifest-load` / `target-discovery` / `target-selection` / `output-planning`，以及 selected package 无 discovered targets 的 `build-plan` 校验，并带出 `stage` 与场景化 detail；build-plan recursion failure 则会稳定落到 `stage = build-plan`，并按场景补 `owner_manifest_path` / `dependency_manifest_path` / `cycle_manifests`，已覆盖 local dependency cycle 与递归 dependency manifest / target-discovery failure；dependency/interface prep failure 则会稳定落到 `stage = dependency-interface-prep`，并按场景补 `owner_manifest_path` / `reference_manifest_path` / `reference` / `failing_dependency_count` / `first_failing_dependency_manifest`，底层若是依赖接口发射失败则继续补 `output_path` / `source_root` / `failing_source_count` / `first_failing_source`；per-target target-prep failure 则会稳定落到 `stage = target-prep`，并按场景补 `dependency_manifest_path` / `dependency_package` / `interface_path` / `symbol` / `first_dependency_package` / `first_dependency_manifest_path` / `conflicting_dependency_package` / `conflicting_dependency_manifest_path` / `io_path` 等 detail；build-side interface emit failure 则会稳定落到 `stage = emit-interface`，并按场景补 `output_path` / `source_root` / `failing_source_count` / `first_failing_source` 等 detail；actual target build failure 则会保留已完成的 partial built_targets，并继续把 `diagnostics` / `invalid-input` / `io` / `toolchain` 这四类 target build failure 结构化到 `failure` 字段。之前剩下的 per-target dependency extern/interface artifact prep 这块失败面，现在也已稳定留在 `ql.build.v1` 的 stdout contract 内，而不会再绕回 stderr-only 退出。
- `ql run` 这轮也已复用同一套 project target 发现面：单文件路径会直接按 `exe` build 并执行，package/workspace 路径默认仍要求全局恰好只有一个 runnable target；当前 runnable 仅包括 `bin` 与 fallback `source`，`lib` 不参与 `run`。如果没有 runnable target，或解析出多个 runnable target，CLI 会显式失败并提示重跑 `ql project targets` 检查发现结果；但现在也可以用 `--package <name>`、`--bin <name>` 或 `--target <path>` 在多 target 项目里稳定选中唯一 runnable target。`ql run` 这轮还会先静默同步 root package 所需的本地依赖 `.qi`，再静默构建 root package 的本地依赖闭包，不把 dependency interface/build 消息混进程序 stdout/stderr。`ql run` 现在也已补上 `--profile <debug|release>`；对于 package/workspace 路径，如果 CLI 没有显式传 `--profile` 或 `--release`，则会按 `package profile -> workspace profile -> debug` 的顺序解析默认 profile。成功执行时会直接透传子进程退出码，并把 stdout/stderr 留给程序本身。
- `ql test` 这轮已补到最小 project-aware harness：单文件路径会直接把该 `.ql` 文件按 executable 构建并执行，package/workspace/`qlang.toml` 路径则会递归发现各 package 下的 `tests/**/*.ql`。其中普通测试文件仍逐个构建成 executable 后执行，并以进程退出码 `0` 视为通过；而 `tests/ui/**/*.ql` 现在会直接按单文件语义分析执行，并要求 diagnostics 输出与相邻 `.stderr` snapshot 完全一致。`--package <name>` 会先把范围收窄到单个 package/workspace member，`--list` 继续只列 discovery 结果而不执行，`--filter <substring>` 继续按 display path 缩小到一组 smoke / UI targets；`--json` 则会把同一套 discovery/执行结果稳定导出成 `ql.test.v1`，覆盖 `listed / ok / failed / no-tests / no-match` 五类状态，并把 build failure 安静收敛进 JSON，不再在成功进入 JSON slice 后混出额外 stderr 噪音。`ql test` 这轮还会在执行 smoke tests 之前静默预构建测试所属 package 的本地依赖闭包，但不会把 dependency build 消息混进正常测试输出；UI snapshot tests 仍只走 analysis/diagnostics。`ql test` 现在也已补上 `--profile <debug|release>`，对 package/workspace 路径里的普通 smoke tests，如果 CLI 没有显式传 `--profile` 或 `--release`，则会按 `package profile -> workspace profile -> debug` 的顺序解析默认 profile。workspace member 没有测试时会保守跳过，整次范围里完全没有发现测试文件时才会显式失败，而 discovery 成功但过滤后没有命中时也会单独显式失败。失败摘要会继续带出失败测试的退出码、stdout/stderr 或 UI snapshot mismatch 明细，而新生成的 package/workspace 脚手架现在也会默认带上 `src/main.ql` 与 `tests/smoke.ql`。
- `qlsp` 这轮补上了四条更接近真实编辑器使用的能力：workspace roots 驱动的 `workspace symbol` 索引，healthy package/workspace 下的 package/workspace import 与当前文件命中的 dependency value / enum variant / struct field / method member里，`hover` 会优先取唯一 workspace 源码定义内容，而 `definition` / `declaration` 会优先跳到对应源码定义、找不到时再回退 `.qi`；如果命中的是 struct / enum / trait / type alias 这类 workspace/package import root，`typeDefinition` 现在也会优先跳到同一份 workspace 源码定义，而 workspace/package import-root 的 `references` 现在也会在当前文件 import occurrences 之外，额外带上 workspace 其他源码文件里的同路径 import occurrences，以及 workspace source 文件里的同文件 occurrences。dependency-backed current-document `references` 在 healthy package/workspace 且能唯一回溯到 workspace 源码目标时，会以当前文件 occurrence 为起点：include declaration 时把声明位点替换成 workspace 源码定义，并额外并入 workspace source 文件里的同文件 occurrences，以及 workspace 其他源码文件里解析到同一 dependency definition target 的 dependency-backed occurrences；include declaration 关闭时也会保留这类源码优先合并，只是不把源码定义本身并回结果。当前回归已锁定 source function / source method / source enum variant / source struct field 四类，但仍不是完整 workspace-wide reference index。current-document `documentHighlight` 复用 same-file / package-aware references 面统一高亮当前文件里的定义与使用位，以及 resolved dependency import roots、dependency value root / enum variant / 显式 struct field label / 唯一 method member 进入 package-aware `semantic tokens`；其中 healthy package path 下的 dependency import roots 现在也会优先覆盖 generic import token 并提升成解析后的真实 symbol kind。这条 semantic-token 保留面现在也已覆盖“当前文件有 parse errors 或 source diagnostics，但 package/dependency 上下文仍可 best-effort 恢复”的场景，不再整份文档直接掉回 TextMate fallback，而且 broken-source fallback 下的 dependency import roots 现在也会按 resolved dependency symbol kind 继续保留；同一条 best-effort fallback 现在也开始保留 dependency import-root 的常用 hover / definition / declaration / references / `typeDefinition` / `prepareRename` / `rename`，而这轮也把 workspace/package import-root 的源码优先 `hover` / `definition` / `declaration` / `references` / `typeDefinition`，以及 current-document `documentHighlight` / `prepareRename` / `rename` 接进了 parse-error fallback；另外，dependency import-backed current-document value root 现在也已进入这条 parse-error 合同，会继续保留源码优先 `hover` / `definition` / `declaration` / `typeDefinition` 与 `references`，也会保留 direct `.field` / `.method()` member completion（现在也覆盖 `load().value` / `load().get(` 这类 import-root 直接调用结果，以及 `maybe_load()?.value` / `maybe_load()?.get(` 这类 question-call receiver）、direct question-unwrapped receiver 的 member completion，并带出 workspace 源码定义与其他 workspace 文件里的同目标 dependency-backed occurrences，而同一条 broken-source fallback 现在也已显式锁住这类 value root 的 current-document `documentHighlight` / `prepareRename` / `rename`；`let current = config.child()?` 这类 local method-result alias root 的 `hover` / `definition` / `references` 与 alias-member completion 也继续沿用同一条 fallback；其中 rename 仍只改当前文档 occurrence，不开放 cross-file rename / workspace edits；在 `include declaration = false` 下会跳过那些 cross-file definition sites。当前这条 parse-error 扩展仍不扩到这些 import-call receiver 的 hover / definition / references、direct question receiver 的 hover / definition / references、indexed/structured receiver，或更宽的 method-result member surface。
- VSCode 插件已补最小 TextMate grammar fallback，因此在 `qlsp` semantic tokens 不足或未返回时，`.ql` 文件至少仍有基础语法着色。
- package/workspace 已支持 manifest 加载、成员遍历、`.qi` 生成/校验/同步和状态展示；`ql project graph`、`ql project emit-interface --check` 现在会解释 `stale` 原因，并给 `invalid` / `unreadable` 补 detail；package 级和 workspace member 级的 `ql project emit-interface --check` 失败现在也会补 `failing package manifest`；workspace 根 `ql project graph` 在单个 member manifest 无法加载时也会继续输出其余 members，并把坏 member 标成 `package: <unresolved>` + `member_status` + `member_error`；其中缺 `[package].name` 会标成 `unresolved-package`，其余 manifest 读取/解析失败会标成 `unresolved-manifest`；如果坏 member 的问题是缺 `[package].name`，`member_error` 现在也会收敛成 `manifest ... does not declare [package].name`，而且这些 `member_error` / `detail` 字段里的 manifest 路径现在也统一按 graph 根目录做相对化，不再混入绝对路径；direct `ql project graph` 命中同一问题时 stderr 仍保留命令标签，而且 direct graph 的 manifest 自身语法损坏、不可读或缺 `[package].name` 时，现在也会补 `failing package manifest` 与直接 rerun hint；如果 direct `ql project graph` 的目标路径根本不在任何 package/workspace manifest 下，现在也会明确说明这条命令只接受可由 `qlang.toml` 发现的 package/workspace 上下文，并给出针对原路径的直接 rerun hint。`ql project graph` 里的 `reference_interfaces` 对 `unresolved-manifest` / `unresolved-package` 现在也会带 `detail`，并且每条 reference 现在都会显式带出对应 manifest 路径；如果 direct dependency 下面还有更深层坏引用，graph 现在也会补 `transitive_reference_failures` 计数，以及第一个坏点的 manifest / `.qi` path / status / detail；如果这个坏点本身是 stale，还会继续补第一处 transitive stale 的 `stale_reasons`；workspace 根 `ql project emit-interface` 在单个 member 发射失败时也会继续输出其余成功 members，并在末尾汇总失败成员数；只有多失败场景才额外补 `first failing member manifest`。如果某个 member 是因为 package 源码错误而发射失败，workspace emit 现在也会当场先补该 member 的 failing package manifest、`failing workspace member manifest`，再给直接重跑 hint；如果某个 member manifest 自身无法加载，workspace emit 和 workspace `emit-interface --check` 的局部错误块现在也都会立即补 `failing workspace member manifest` 和针对该 member manifest 的直接 rerun hint；如果某个 member
